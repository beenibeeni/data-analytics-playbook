Module 1 · Querying basics

SELECT and column basics

Every SQL query starts the same way: say what columns you want, and where to get them from.

1
The SELECT … FROM skeleton
Core syntax
▼

Read this out loud as "give me these columns, from this table" — that's literally what it does.

Copy 
SELECT job_title, company, work_setup
FROM job_postings;
  1. List the exact column names after SELECT, separated by commas.
  2. Name the table after FROM.
  3. End the statement with a semicolon — some tools don't require it, but get the habit early.
📝
SQL keywords (SELECT, FROM) are not case-sensitive, but writing them in CAPS while keeping column/table names lowercase is the universal convention — it makes long queries scannable at a glance.
2
Selecting everything, renaming columns, removing duplicates
3 variations
▼
Copy 
SELECT *
FROM job_postings;          -- every column, every row

SELECT job_title AS title,
       work_setup AS setup
FROM job_postings;          -- AS renames a column in the output

SELECT DISTINCT work_setup
FROM job_postings;          -- unique values only
DISTINCT work_setup → output
Remote · Hybrid · Onsite
⚠️
SELECT * is fine for exploring a table you don't know yet, but avoid it in any query you'll reuse or build a report on — it silently breaks if someone adds a column later, and it's slower on large tables.

Filtering rows with WHERE

WHERE is how you go from "all the rows" to "only the rows I actually need."

1
Comparison & logical operators
Core syntax
▼
OperatorMeaning
= <> / !=equal to / not equal to
> < >= <=greater/less than (works on numbers and dates)
AND / ORcombine multiple conditions
BETWEEN a AND binclusive range
IN (a, b, c)matches any value in a list
LIKEpattern match on text, using % as a wildcard
Copy 
SELECT job_title, salary_midpoint
FROM job_postings
WHERE work_setup = 'Remote'
  AND salary_midpoint >= 50000;

SELECT job_title
FROM job_postings
WHERE job_title LIKE '%Analyst%';   -- title contains "Analyst" anywhere
💡
Always wrap text values in single quotes ('Remote'), never double quotes. Double quotes in most SQL dialects refer to a column or table name, not a literal value — a very common beginner error message.

Module 1 · Querying basics

Sorting and limiting results

ORDER BY controls the sequence rows come back in; LIMIT controls how many you see.

1
ORDER BY and LIMIT / TOP
Core syntax
▼
Copy 
SELECT job_title, salary_midpoint
FROM job_postings
ORDER BY salary_midpoint DESC   -- highest first; ASC is the default
LIMIT 5;                        -- top 5 rows only
📝
SQL Server uses SELECT TOP 5 ... instead of LIMIT 5 at the end. PostgreSQL, MySQL, and SQLite all use LIMIT. Same idea, different keyword placement.

You can sort by more than one column — the second column only breaks ties left by the first.

Copy 
ORDER BY work_setup ASC, salary_midpoint DESC

Module 1 · Querying basics

Handling NULLs

NULL means "no value recorded" — it is not zero, not an empty string, and it breaks ordinary = comparisons. This trips up almost everyone at first.

1
Why "= NULL" never works
Concept
▼

WHERE company = NULL always returns zero rows, even on a row where company really is blank — NULL means "unknown," and SQL won't claim two unknowns are equal. You must use the dedicated NULL-checking syntax instead.

Copy 
SELECT *
FROM job_postings
WHERE company IS NULL;       -- correct way to find blanks

SELECT *
FROM job_postings
WHERE company IS NOT NULL;   -- correct way to exclude blanks
⚠️
NULLs are also skipped silently by aggregate functions like AVG() — that's usually what you want (NULL salaries shouldn't drag the average down to zero), but always double-check whether "no value" should be treated as zero or genuinely excluded for your specific metric.

Aggregate functions

These collapse many rows into one summary number — the SQL equivalent of a Power BI card visual.

1
COUNT, SUM, AVG, MIN, MAX
5 functions
▼
FunctionReturns
COUNT(*)number of rows, NULLs included
COUNT(column)number of non-NULL values in that column
SUM(column)total of a numeric column
AVG(column)average, ignoring NULLs
MIN(column) / MAX(column)smallest / largest value
Copy 
SELECT
    COUNT(*) AS total_jobs,
    AVG(salary_midpoint) AS avg_salary,
    MAX(salary_midpoint) AS highest_salary
FROM job_postings;
Output (one row)
total_jobs: 100  |  avg_salary: 84,210  |  highest_salary: 250,000

Module 2 · Aggregating

GROUP BY

GROUP BY is what turns "one big average" into "average per category" — this is the single most useful clause for BI-style reporting.

1
The rule that confuses everyone at first
Concept + syntax
▼

Every column in your SELECT list must be either (a) inside an aggregate function, or (b) listed in GROUP BY. SQL needs to know exactly what to do with every column for every group — there's no "just show me one of them" option.

Copy 
SELECT
    work_setup,
    COUNT(*) AS job_count,
    AVG(salary_midpoint) AS avg_salary
FROM job_postings
GROUP BY work_setup
ORDER BY job_count DESC;
Output
work_setupjob_countavg_salary
Remote5489,400
Hybrid3081,200
Onsite1676,900
💡
Mental model: GROUP BY first splits the table into mini-tables, one per distinct value — then runs your aggregate functions separately on each mini-table, and stitches the one-row results back together.
2
Grouping by more than one column
1 example
▼

List multiple columns to create a group per unique combination — for example, work setup crossed with employment type.

Copy 
SELECT work_setup, employment_type, COUNT(*) AS job_count
FROM job_postings
GROUP BY work_setup, employment_type;

Module 2 · Aggregating

Filtering groups with HAVING

WHERE filters individual rows before grouping; HAVING filters the groups after they've been built. Mixing these up is the #1 GROUP BY error.

1
WHERE vs HAVING, side by side
Core syntax
▼
Copy 
SELECT work_setup, COUNT(*) AS job_count
FROM job_postings
WHERE salary_midpoint IS NOT NULL   -- filters rows, runs BEFORE grouping
GROUP BY work_setup
HAVING COUNT(*) > 10;             -- filters groups, runs AFTER grouping
WHEREHAVING
Filtersindividual rowsaggregated groups
Runsbefore GROUP BYafter GROUP BY
Can reference an aggregate like COUNT(*)?NoYes
⚠️
Writing WHERE COUNT(*) > 10 will throw an error in most databases — at the point WHERE runs, the rows haven't been grouped yet, so there's no count to check.

Module 3 · Combining tables

Why joins exist

Real databases split data across multiple tables on purpose — one row per job posting in one table, one row per application in another. A join is how you bring related rows back together for a query.

1
The shared key that makes a join possible
Concept
▼

A join works because both tables share a common column — usually an ID. In this project, job_postings.job_id and applications.job_id are the same value, so SQL can line up matching rows from each table.

📝
This is exactly the relationship you'd draw as a line between two tables in Power BI's Model view — a join in SQL and a relationship in Power BI are the same underlying concept, just expressed differently.
← BackNext: INNER JOIN →
Module 3 · Combining tables

INNER JOIN

Returns only the rows that have a match in both tables. This is the join you reach for by default.

1
Syntax and what gets dropped
Core syntax
▼
Copy 
SELECT
    jp.job_title,
    jp.work_setup,
    a.gender,
    a.interview_score
FROM job_postings AS jp
INNER JOIN applications AS a
    ON jp.job_id = a.job_id;
  1. Alias each table (jp, a) so you can prefix columns and avoid ambiguity — required once two tables have a same-named column like job_id.
  2. ON defines the matching condition — almost always an ID = ID comparison.
  3. Any job posting with zero applications, and any application with a job_id that doesn't exist in job_postings, is silently excluded from the result.
⚠️
If your row count after a join looks lower than expected, this dropping behavior is almost always why. Before joining, it's worth knowing roughly how many rows should have a match on each side.
← BackNext: LEFT JOIN →
Module 3 · Combining tables

LEFT JOIN

Keeps every row from the left (first) table, whether or not it has a match — filling in NULLs where there's nothing to match. Use this when "no match" is itself meaningful information.

1
Finding job postings with zero applications
Real use case
▼

This is the classic LEFT JOIN use case: keep every job posting, attach applications where they exist, then filter for the ones where nothing attached.

Copy 
SELECT jp.job_id, jp.job_title
FROM job_postings AS jp
LEFT JOIN applications AS a
    ON jp.job_id = a.job_id
WHERE a.application_id IS NULL;   -- no matching application row
💡
Picture it as: keep the whole left table intact, and just bolt on whatever fits from the right table — empty seats become NULL instead of disappearing. INNER JOIN would have silently dropped every one of these postings.

Module 3 · Combining tables

Joining three or more tables

You can chain joins — each new JOIN clause just adds another table to whatever you've already built.

1
Chaining joins, e.g. with a dim_skills table
Pattern
▼
Copy 
SELECT jp.job_title, a.gender, s.skill_name
FROM job_postings AS jp
INNER JOIN applications AS a  ON jp.job_id = a.job_id
LEFT JOIN  job_skills  AS js ON jp.job_id = js.job_id
LEFT JOIN  dim_skills  AS s  ON js.skill_id = s.skill_id;
📝
This mirrors the star schema you'd build in Power BI's Model view — a fact table (job_postings) joined out to bridge and dimension tables (job_skills, dim_skills). Each JOIN line is one relationship line in the diagram.

Module 4 · Going further

Subqueries

A query nested inside another query — useful whenever you need a calculated value before you can filter or compare against it.

1
Filtering against a value the database has to calculate first
Example
▼

"Show me jobs paying above average" requires knowing the average first — you can't hardcode it, because it changes as data changes. A subquery computes it inline.

Copy 
SELECT job_title, salary_midpoint
FROM job_postings
WHERE salary_midpoint > (
    SELECT AVG(salary_midpoint) FROM job_postings
);
💡
The database runs the inner query first (it returns one number), then plugs that number into the outer query as if you'd typed it in by hand.

Module 4 · Going further

CTEs — the WITH clause

A Common Table Expression names a subquery and lets you reference it like a temporary table — the same logic as a subquery, but readable top-to-bottom instead of nested inside-out.

1
Rewriting the subquery above as a CTE
Same logic, cleaner shape
▼
Copy 
WITH avg_salary AS (
    SELECT AVG(salary_midpoint) AS overall_avg
    FROM job_postings
)
SELECT jp.job_title, jp.salary_midpoint
FROM job_postings AS jp, avg_salary
WHERE jp.salary_midpoint > avg_salary.overall_avg;
📝
CTEs really pay off once you need the same intermediate result more than once in a query, or you're chaining several calculation steps — you can stack multiple WITH ... AS (...) blocks, comma-separated, before the final SELECT.

Module 4 · Going further

Window functions

These calculate something across a set of rows — a rank, a running total, a comparison to the group average — without collapsing the rows the way GROUP BY does. You keep every row and get an extra calculated column.

1
RANK() within a group, using OVER (PARTITION BY ...)
The pattern to memorize
▼
Copy 
SELECT
    job_title,
    work_setup,
    salary_midpoint,
    RANK() OVER (
        PARTITION BY work_setup
        ORDER BY salary_midpoint DESC
    ) AS rank_in_setup
FROM job_postings;
Output (rank resets per work_setup group)
job_titlework_setupsalary_midpointrank_in_setup
Senior Data EngineerRemote180,0001
Data AnalystRemote62,0002
AI/ML EngineerHybrid150,0001
💡
PARTITION BY is "GROUP BY, but don't collapse the rows." It resets the calculation for each group while still showing you every individual row alongside it — exactly what you'd need for a "top 3 highest paying jobs per category" report.

Module 4 · Going further

CASE WHEN logic

SQL's version of an if/else statement — this is exactly the logic behind the age_band custom column from the Power Query guide, written in SQL instead.

1
Building a bucketed column
1 example
▼
Copy 
SELECT
    age,
    CASE
        WHEN age IS NULL THEN NULL
        WHEN age < 25 THEN 'Under 25'
        WHEN age < 35 THEN '25-34'
        WHEN age < 45 THEN '35-44'
        ELSE '45+'
    END AS age_band
FROM applications;
⚠️
CASE checks conditions top to bottom and stops at the first match — so order them from most specific to least specific, the same way you ordered the date-parsing formula's else if chain in Power Query.

Module 5 · Review

Query execution order (not the order you write it in)

This single fact explains almost every "why doesn't this work" moment in SQL — the order you type clauses is not the order the database actually runs them.

1
Written order vs. logical execution order
The most important table in this guide
▼
You write it asThe database runs it as
1. SELECT1. FROM / JOIN
2. FROM2. WHERE
3. JOIN3. GROUP BY
4. WHERE4. HAVING
5. GROUP BY5. SELECT
6. HAVING6. ORDER BY
7. ORDER BY7. LIMIT
📝
This is why WHERE can't reference an aggregate (it runs before grouping happens), why a column alias from SELECT can't be used in WHERE (the alias doesn't exist yet at that stage), and why ORDER BY can reference an alias (by the time it runs, SELECT has already created it).
